Central DB Sync — Data Schema Reference

PostgreSQL · schemas: sync_meta, report · Phân tích chi tiết từng bảng, field, luồng dữ liệu và tương tác trong sync engine

I. Tổng quan

Central DB Sync sử dụng 2 schemas trên PostgreSQL, tổng cộng 4 bảng:

SchemaTableLoạiMục đích
sync_metatable_sync_config Config Đăng ký những table nào được sync, cấu hình sync mode, tier, dependency
sync_metacheckpoint Checkpoint Trạng thái sync hiện tại (version, status, health) cho mỗi table
sync_metasync_run_log Audit log Append-only log — mỗi sync run là 1 row
reportpartners Target data Dữ liệu partner đã sync từ ERP, dùng cho reporting
Quan hệ: checkpoint.source_table → FK → table_sync_config.source_table. sync_run_log.source_table là logical FK (không enforced).

II. sync_meta.table_sync_config

sync_meta.table_sync_config

Config registry

Sổ đăng ký — khai báo những table nào được sync engine quản lý. Mỗi table chỉ có 1 row. Dữ liệu được seed thủ công qua SQL script, không tự động insert từ code.

Field definitions:

ColumnTypeRequiredDefaultMô tảGhi chú
source_table TEXT PK — Tên table trên MSSQL source, VD: CRM.Partners Phải match chính xác tên table ở ERP. Không alias — checkpoint keys dùng giá trị này.
target_schema TEXT ✅ 'report' Schema đích trên PostgreSQL
target_table TEXT ✅ — Tên table đích trên PostgreSQL, VD: partners
sync_mode TEXT ✅ 'ChangeTracking' Phương thức sync: ChangeTracking (CT incremental) hoặc FullRefresh (full table scan mỗi lần) Bị ràng buộc CHECK. Pilot dùng ChangeTracking.
sync_tier TEXT ✅ 'Hot' Mức độ ưu tiên sync: Hot (sync thường xuyên) hoặc Cold (ít chạy) Pilot dùng Hot.
dependency TEXT[] ❌ '{}' Danh sách table phải sync xong trước table này. Ví dụ: ['CRM.Partners'] SyncOrchestrator kiểm tra dependency trước khi chạy. Partners hiện không có dependency.
expected_sync_interval INTERVAL ✅ '1 minute' Tần suất sync kỳ vọng. Dùng để tính lag tolerance.
max_allowed_lag INTERVAL ✅ '5 minutes' Thời gian tối đa cho phép từ lần sync cuối. Vượt quá → Degraded. Health endpoint dùng giá trị này làm default nếu không truyền.

III. sync_meta.checkpoint

sync_meta.checkpoint

Sync progress

Trái tim của sync engine — lưu trạng thái sync hiện tại. Mỗi table có 1 row. Được đọc/ghi bởi hầu hết các component.

Field definitions:

ColumnTypeRequiredDefaultMô tảGhi chú
source_table TEXT PK FK → table_sync_config — Tên table, VD: CRM.Partners FK đảm bảo không thể có checkpoint cho table chưa đăng ký.
last_sync_version BIGINT ❌ NULL Database-wide CT watermark cuối cùng mà table này đã xử lý thành công. NULL nếu chưa bootstrap. Giá trị này là baseline cho CHANGETABLE query của riêng source_table ở lần sync tiếp theo. Có thể tăng dù table đó không có row thay đổi.
sync_status TEXT ✅ 'pending_initial_sync' Trạng thái hiện tại pending_initial_sync — chờ bootstrap lần đầu
ready — đã sync thành công, sẵn sàng cho CT incremental
requires_full_resync — cần bootstrap lại (do lỗi)
last_attempt_at TIMESTAMPTZ ❌ NULL Thời điểm sync lần gần nhất (cả thành công lẫn thất bại) Set bởi TransitionToFullResyncAsync khi error. Không set bởi bootstrap/CT success path.
last_success_at TIMESTAMPTZ ❌ NULL Thời điểm sync thành công cuối cùng Set bởi cả bootstrap và CT. Dùng để tính lag (so với current time).
last_failure_at TIMESTAMPTZ ❌ NULL Thời điểm sync thất bại gần nhất Set bởi TransitionToFullResyncAsync.
consecutive_failure_count INTEGER ✅ 0 Số lần sync fail liên tiếp. Reset về 0 khi sync thành công. Dùng trong health check: > 0 → Degraded. Tăng dần mỗi lần TransitionToFullResyncAsync.
last_error_code TEXT ❌ NULL Error code của lần fail gần nhất. Null khi thành công. VD: BootstrapFailed, ApplyRetriesExhausted, CONCURRENT_CHECKPOINT
last_error_message TEXT ❌ NULL Chi tiết lỗi (exception message, inner exception)

Semantics của last_sync_version:

CT version thuộc phạm vi SQL Server database, không thuộc riêng từng table. CHANGE_TRACKING_CURRENT_VERSION() trả về một watermark chung cho toàn database có Change Tracking. Vì vậy, thay đổi trên TableA có thể làm current version tăng từ 100 lên 101, dù TableB không có row nào thay đổi.
Job TableA → đọc CHANGETABLE(CHANGES TableA, 100) → có rows → update checkpoint của TableA thành 101
Job TableB → đọc CHANGETABLE(CHANGES TableB, 100) → không có rows → vẫn update checkpoint của TableB thành 101
Điểm quan trọng: thay đổi trên một source table không trực tiếp update checkpoint của các table khác. Mỗi sync job chạy độc lập và câu UPDATE luôn có WHERE source_table = @sourceTable AND last_sync_version = @previousCheckpoint. Các checkpoint trông giống như được cập nhật cùng lúc vì từng job đều đọc cùng database-wide upper watermark, rồi tự advance row của chính nó, kể cả khi kết quả là no_changes.
Không nên diễn giảiNên diễn giải
last_sync_version = 101 nghĩa là table có row thay đổi ở version 101 Job của table đã kiểm tra và xử lý an toàn mọi CT event của table đó đến hết watermark 101
Hai table có cùng checkpoint nghĩa là dữ liệu hai table thay đổi cùng lúc Hai table đã được xử lý đến cùng một mốc trong timeline CT chung của database
Checkpoint nhảy qua nhiều version là mất dữ liệu Các version ở giữa có thể thuộc table khác; CHANGETABLE vẫn lọc event theo table đang sync

Luồng thay đổi trạng thái (state machine):

┌─────────────────────────────────────────────────────────────────┐
│                      STATE TRANSITIONS                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                   
    -- Seed:   pending_initial_sync  (last_sync_version = NULL)
         │
         ▼  (bootstrap thành công)
    ready  (last_sync_version = baseline CT version)
         │
         ├── (CT sync thành công) → ready  (last_sync_version tăng)
         │
         ├── (CT lỗi) → requires_full_resync  (consecutive_failure_count + 1)
         │       │
         │       ▼  (bootstrap lại)
         │    ready
         │
         └── (bootstrap lỗi) → requires_full_resync
                  │
                  ▼  (bootstrap lại)
               ready
└─────────────────────────────────────────────────────────────────┘

Tương tác với các thành phần:

PostgresSyncCheckpointStore
• GetAsync() — đọc checkpoint để quyết định đường đi
• AdvanceAsync() — CT sync: update version + reset fail count
• TransitionToFullResyncAsync() — lỗi: set status + ghi error metadata
PostgresPartnersWriter
• Bootstrap: UPDATE checkpoint SET version + status = 'ready' (trong cùng transaction với data upsert)
• CT batch: UPDATE checkpoint SET version (optimistic guard trong cùng transaction)
CentralDbSyncHealthService
• Đọc checkpoint → quyết định Healthy / Degraded / Unknown
SyncOrchestrator
• Đọc checkpoint → quyết định route: bootstrap hay CT service
ChangeTrackingSyncService
• Validate checkpoint null/status/version → RequiresFullResync hoặc ReadBatch
📍 Checkpoint là row duy nhất (1 per table). Tất cả thao tác đều là UPDATE, không có INSERT mới (trừ seed).

IV. sync_meta.sync_run_log

sync_meta.sync_run_log

Audit trail

Append-only audit log. Mỗi sync run (bootstrap, CT, orchestrator decision) ghi 1 row. Không bao giờ UPDATE hay DELETE — chỉ INSERT. Dùng để debug, monitoring, trace lịch sử.

Field definitions:

ColumnTypeRequiredDefaultMô tảGhi chú
id BIGSERIAL PK auto Auto increment — thứ tự log
source_table TEXT ✅ — Tên table, VD: CRM.Partners Logical FK tới table_sync_config (không enforced vì là append log).
run_id UUID ✅ — UUID xác định duy nhất 1 lần chạy. Dùng để trace xuyên suốt.
mode TEXT ✅ — Loại sync run: Bootstrap, ChangeTracking, Orchestrator Orchestrator mode chỉ log khi skip dependency; không log mỗi lần orchestrate.
outcome TEXT ✅ — Kết quả sync run. Bị ràng buộc CHECK: succeeded — thành công
no_changes — CT không có thay đổi
failed — lỗi
skipped_locked — lock busy
skipped_dependency — dependency chưa ready
requires_full_resync — checkpoint invalid
rows_read INTEGER ✅ 0 Số rows đọc được từ MSSQL (bootstrap: tổng rows, CT: số changes)
rows_upserted INTEGER ✅ 0 Số rows INSERT/UPDATE thành công trên PostgreSQL
rows_deactivated INTEGER ✅ 0 Số rows bị SET is_active = false (bootstrap: orphan rows, CT: delete + non-customer)
rows_deleted INTEGER ✅ 0 Số rows bị xóa vật lý (hiện tại pilot không dùng — CT delete chuyển thành deactivate)
checkpoint_before BIGINT ❌ NULL CT version trước khi sync
checkpoint_after BIGINT ❌ NULL CT version sau khi sync
started_at TIMESTAMPTZ ✅ NOW() Thời điểm bắt đầu sync run
finished_at TIMESTAMPTZ ❌ NULL Thời điểm kết thúc
duration_ms INTEGER ❌ NULL Thời gian chạy (milliseconds)
error_code TEXT ❌ NULL Mã lỗi nếu outcome = 'failed'. VD: BootstrapFailed, ApplyRetriesExhausted
error_message TEXT ❌ NULL Chi tiết lỗi (exception message)

Index:

CREATE INDEX idx_sync_run_log_table_started
    ON sync_meta.sync_run_log (source_table, started_at DESC);

Index này tối ưu query: "cho tôi xem lịch sử sync gần nhất của table X" — query thường gặp khi debug.

Luồng ghi log:

BootstrapSyncService / ChangeTrackingSyncService / SyncOrchestrator
│ ├─ (sau khi sync hoàn tất) → PostgresSyncRunLog.WriteAsync() │ INSERT INTO sync_run_log (mode, outcome, rows_read, ..., duration_ms) │ └─ (log nằm NGOÀI transaction apply data) → Nếu apply thành công mà log fail: dữ liệu đã sync, checkpoint đã update, nhưng không có trace. Acceptable risk. → Nếu apply fail mà log thành công: Log vẫn tồn tại để debug, checkpoint không đổi.
📍 Log là append-only, không có cơ chế cleanup. Cần monitor size theo thời gian. Với interval 1 phút: ~1,440 rows/ngày, ~525k rows/năm — rất nhỏ.

V. report.partners

report.partners

Target data

Bảng dữ liệu đích — denormalized copy của [dbo].[CRM.Partners] từ ERP. Được sync engine duy trì. Các application query từ bảng này thay vì query trực tiếp ERP.

Field definitions:

ColumnTypeRequiredDefaultMô tảGhi chú
partner_id INTEGER PK — Khóa chính — business key từ ERP
company_id INTEGER ✅ — Công ty sở hữu partner này
code TEXT ✅ — Mã partner
name TEXT ✅ — Tên partner
is_customer BOOLEAN ✅ FALSE Flag từ ERP: partner này có phải khách hàng không Quan trọng: is_active = is_customer. Nếu ERP set is_customer = false → bị deactivate.
is_supplier BOOLEAN ✅ FALSE Flag từ ERP: partner này có phải nhà cung cấp không
email TEXT ❌ NULL
phone TEXT ❌ NULL
activated BOOLEAN ✅ TRUE Trạng thái activated từ ERP Không liên quan is_active. Đây là field gốc từ ERP.
is_active BOOLEAN ✅ TRUE Sync engine control flag. FALSE = partner đã bị xóa/deactivate trên ERP hoặc không còn là customer Field quan trọng nhất. Application query nên filter WHERE is_active = true. Không phải field từ ERP — do sync engine tự quản lý.
synced_at TIMESTAMPTZ ✅ NOW() Thời điểm sync gần nhất cho row này
source_system TEXT ✅ 'erp' Nguồn dữ liệu. Cho phép multiple sources trong tương lai. Dùng trong bootstrap deactivate: WHERE source_system = 'erp' AND partner_id != ALL(@snapshotPks)

Logic is_active:

Bootstrap
is_active = COALESCE(EXCLUDED.is_customer, false)
→ partner được set active chỉ khi is_customer = true trên ERP
CT incremental
• Operation = 'D' (deleted) → is_active = false
• IsCustomer != true → is_active = false
• IsCustomer == true → upsert với is_active = true
Bootstrap deactivate orphan
UPDATE report.partners SET is_active = false WHERE source_system = 'erp' AND partner_id != ALL(@snapshotPks)
→ Nếu partner không còn trong ERP snapshot → bị deactivate
📍 Không có xóa vật lý. partners chỉ được deactivate (is_active = false), không bao giờ DELETE. Đảm bảo dữ liệu lịch sử vẫn còn để truy vấn.

Tương tác với các thành phần:

PostgresPartnersWriter
• ApplyBootstrapAsync — upsert toàn bộ snapshot + deactivate orphan + set checkpoint (cùng 1 transaction)
• ApplyBatchAsync — deactivate delete + non-customer rows + upsert active customer rows + advance checkpoint (cùng 1 transaction)
Application / Reporting
• Query report.partners WHERE is_active = true để lấy danh sách customer hiện tại
• Dùng cho Base44, Retool, hoặc bất kỳ internal service nào cần dữ liệu partner
📍 report.partners không có foreign key đến sync_meta — độc lập. Sync engine có thể bị tắt mà không ảnh hưởng đến application query (dữ liệu chỉ stale, không mất).

VII. Data Flow — Tổng quan

Luồng ghi:

MSSQL (ERP) → Reader đọc data → Writer upsert report.partners + update checkpoint + insert sync_run_log

Luồng đọc:

SyncOrchestrator → đọc table_sync_config (biết table nào enabled, dependency gì)
SyncOrchestrator → đọc checkpoint (biết trạng thái → quyết định bootstrap hay CT)
HealthService → đọc checkpoint (quyết định Healthy/Degraded/Unknown)
Application → đọc report.partners (lấy dữ liệu reporting)
Debug → đọc sync_run_log (trace lịch sử sync)

Luồng seed (1 lần khi deploy):

002-central-db-sync-seed.sql → INSERT table_sync_config + INSERT checkpoint (pending_initial_sync)
