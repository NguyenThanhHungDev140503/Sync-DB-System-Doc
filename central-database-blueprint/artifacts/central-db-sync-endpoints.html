Kiểm tra checkpoint freshness / sync lag của một table

Mục đích: Cho phép monitoring (Base44, Retool) kiểm tra trạng thái sync engine mà không cần query trực tiếp PostgreSQL sync_meta.checkpoint.

Lưu ý: Endpoint này không kiểm tra Change Tracking trên MSSQL — nó chỉ đọc sync_meta.checkpoint để đo checkpoint freshness. CT health cần kiểm tra riêng qua MSSQL sys.change_tracking_tables.

Logic xử lý (CentralDbSyncStatusService):

GetStatusAsync(sourceTable, maxAllowedLag):
  checkpoint = ISyncCheckpointStore.GetAsync(sourceTable)
  if (checkpoint == null) → Unknown
  if (checkpoint.SyncStatus == "pending_initial_sync") → Unknown
  if (checkpoint.SyncStatus == "requires_full_resync"
      || checkpoint.ConsecutiveFailureCount > 0) → Degraded
  if (now - lastSuccessAt > maxAllowedLag) → Degraded
  → Healthy

pending_initial_sync trả về Unknown vì chưa có sync nào — hệ thống đang chờ bootstrap lần đầu.

Dependencies: ISyncCheckpointStore → PostgresSyncCheckpointStore (đọc sync_meta.checkpoint).

POST /api/central-db-sync/bootstrap/{sourceTable} HTTP endpoint

Kích hoạt bootstrap (full sync) thủ công

⚠️ Thuật ngữ "snapshot" trong tài liệu này chỉ "bản sao dữ liệu tại một thời điểm", hoàn toàn KHÔNG liên quan đến SQL Server Snapshot Isolation. Hệ thống dùng ReadCommitted + READPAST + CT version verify — không cần bật Snapshot Isolation trên database.

Bootstrap là gì? Bootstrap (full sync) là quá trình đọc toàn bộ dữ liệu hiện tại từ MSSQL source table và ghi đè lên PostgreSQL target table. Không giống CT incremental sync (chỉ sync thay đổi), bootstrap xóa sạch trạng thái cũ và xây dựng lại từ đầu.

Dùng để làm gì?
• 🆕 Khởi tạo lần đầu: Khi một table lần đầu được đưa vào sync engine lần đầu, không có dữ liệu nào trên PostgreSQL và checkpoint null. CT incremental sync không thể tự tạo dữ liệu từ đầu vì nó chỉ đọc thay đổi dựa trên version — không có version cũ nào để so sánh. Bootstrap là cách duy nhất để: (a) có một bản sao dữ liệu đầy đủ làm baseline, (b) ghi checkpoint version đầu tiên, (c) từ đó CT incremental mới có điểm xuất phát để chạy.
• 🔄 Recovery sau lỗi: Khi CT incremental sync bị lỗi nghiêm trọng (checkpoint invalid, retention cleanup), hệ thống tự động chuyển sang bootstrap để đồng bộ lại từ đầu.
• 🛠️ Can thiệp thủ công: Admin gọi POST /bootstrap khi cần force resync (sửa data corruption, sync bị trễ quá lâu, thay đổi mapping).

Cơ chế: Đọc CT version làm baseline → full table scan với READPAST → verify không có concurrent DML (so sánh CT version trước/sau) → upsert tất cả vào PostgreSQL trong 1 transaction → set checkpoint. Toàn bộ dữ liệu cũ bị deactivate nếu không còn trong lần chạy bootstrap hiện tại.

Nếu không chạy bootstrap thì CT có chạy được không? Không. CT incremental sync phụ thuộc hoàn toàn vào checkpoint để hoạt động. Khi chưa bootstrap: (1) checkpoint null hoặc pending_initial_sync → CT service kiểm tra và trả về RequiresFullResync ngay, (2) PostgreSQL chưa có row nào → CHANGETABLE chỉ trả về partner_id không đủ để upsert, (3) SyncOrchestrator tự động route sang bootstrap khi thấy checkpoint chưa ready. Bootstrap là bắt buộc để tạo baseline dữ liệu và checkpoint trước khi CT có thể chạy.

Mục đích: Cho phép admin kích hoạt full re-sync khi cần (sửa lỗi dữ liệu, recovery sau downtime dài).

BootstrapSyncService.ExecuteAsync() — Chi tiết luồng:

1. Acquire lock
ITableSyncLock.TryAcquireAsync() → PostgreSQL pg_try_advisory_lock(). Nếu lock bận → SkippedLocked.
2. Đọc toàn bộ dữ liệu từ MSSQL
IBootstrapSnapshotReader.ReadAsync() → SqlServerPartnersReader: Capture CT version + full table scan với READPAST + verify version (retry 3 lần).
3. Apply to PostgreSQL
ISyncBatchApplier.ApplyBootstrapAsync() → PostgresPartnersWriter: 1 transaction upsert tất cả rows + deactivate orphan + set checkpoint → ready.
4. Log + release lock
ISyncRunLog.WriteAsync() → sync_meta.sync_run_log + connection disposed → lock tự release.

⏱ SyncOrchestrator.ExecuteAsync() Orchestrator

Bộ điều phối sync engine — quyết định bootstrap hay CT sync

Mục đích: Điều hướng luồng sync dựa trên trạng thái checkpoint. Xử lý dependency giữa các table.

Luồng xử lý (per table):

foreach config in configs:
  if (!config.Enabled) → continue
  if (!AreDependenciesReady(config)) → continue (log skipped_dependency)
  checkpoint = checkpointStore.GetAsync(config.SourceTable)

  if (checkpoint == null
      || checkpoint.SyncStatus == "pending_initial_sync"
      || checkpoint.SyncStatus == "requires_full_resync")
    → bootstrapService.ExecuteAsync(config)   -- Bootstrap path
  else
    → ctService.ExecuteAsync(config)            -- CT Incremental path

Dependency check:

CRM.Partners: Không có dependency → luôn được xử lý ngay. Khi mở rộng, table khác (ví dụ: partners_addresses) có thể có dependency là CRM.Partners.

Skipped dependency log:

sync_run_log:
  mode: "Orchestrator"
  outcome: "skipped_dependency"
  duration_ms: 0
  source_table: "TableB"
  error_message: (dependency TableA not Ready)
Error isolation: Mỗi table xử lý độc lập. TableA fail không block TableB (nếu TableB không phụ thuộc TableA).
⏱ BootstrapSyncService.ExecuteAsync() Full sync service

Full snapshot sync — bootstrap toàn bộ dữ liệu từ MSSQL → PostgreSQL

Khi nào được gọi?

  • Endpoint POST /bootstrap
  • SyncOrchestrator khi checkpoint null / pending_initial_sync / requires_full_resync
  • Recovery path: ChangeTrackingSyncService khi checkpoint invalid

Public methods:

MethodMục đích
ExecuteAsync()Standard flow: acquire lock → execute core → lock tự release qua await using
ExecuteWithProvidedLockAsync()CT invalid recovery: execute core với lock đã được acquire trước đó bởi ChangeTrackingSyncService

Chi tiết luồng core:

ExecuteCoreAsync(config, startedAt):
  try
    snapshot = IBootstrapSnapshotReader.ReadAsync(config)
       // → CT baseline version + full data
    result = ISyncBatchApplier.ApplyBootstrapAsync(config, snapshot)
       // → PostgreSQL 1 transaction: upsert + deactivate + checkpoint
    ISyncRunLog.WriteAsync(outcome: succeeded, ...)
    return result
  catch (OperationCanceledException)
    → log Cancelled, throw
  catch (Exception ex)
    → checkpointStore.TransitionToFullResyncAsync()  -- set status + error metadata
    → ISyncRunLog.WriteAsync(outcome: failed, errorCode: "BootstrapFailed")
    return SyncRunResult { Outcome: Failed, ErrorCode: "BootstrapFailed" }
Không re-throw exception sau catch. Bootstrap service luôn trả về SyncRunResult — không crash orchestrator.
⏱ ChangeTrackingSyncService.ExecuteAsync() CT incremental service

Change Tracking incremental sync — chỉ sync dữ liệu thay đổi từ checkpoint

Điều kiện chạy:

Điều kiệnKết quả
Checkpoint null→ RequiresFullResync
SyncStatus != ready→ RequiresFullResync
LastSyncVersion null→ RequiresFullResync
Checkpoint valid + có changes→ Read + apply batch
Checkpoint valid + không changes→ Advance checkpoint, log NoChanges

Error handling:

CheckpointInvalidException: CT retention cleanup → CHECKPOINT_MIN_VALID_VERSION > last_sync_version. Transition → ExecuteWithProvidedLockAsync (bootstrap recovery cùng lock).

Apply retry logic:

ApplyWithRetryAsync(config, batch):
  for attempt = 1..3:
    try
      return ISyncBatchApplier.ApplyBatchAsync(config, batch)
    catch (transient error) when attempt < 3:
      await Delay(attempt^2 seconds)  -- 1s, 2s, 4s
    catch (transient error) when attempt == 3:
      return SyncRunResult { Outcome: Failed, ErrorCode: "ApplyRetriesExhausted" }
    catch (non-transient error):
      return SyncRunResult { Outcome: Failed, ErrorCode: "NonTransientApplyError" }

-- Transient detection:
IsTransient(): message contains "deadlock" | "timeout" | "connection"
No-changes optimization: batch.Rows.Count == 0 → chỉ advance checkpoint, không cần PostgreSQL data transaction. Giảm tải DB.

III. Infrastructure Components

SqlServerPartnersReader MSSQL data reader

Triển khai 3 interfaces: IBootstrapSnapshotReader, IFullRefreshReader, IChangeTrackingReader

1. Bootstrap reader — đọc toàn bộ dữ liệu + CT version baseline

BEGIN TRANSACTION;  -- ReadCommitted
  @baseline = CHANGE_TRACKING_CURRENT_VERSION()
  SELECT {columns} FROM [dbo].[CRM.Partners] WITH (READPAST)
  @version_after = CHANGE_TRACKING_CURRENT_VERSION()
  if (@baseline == @version_after) → COMMIT, return data
  else → ROLLBACK, retry (max 3)
-- Không có COMMIT sau loop — throw InvalidOperationException nếu retry hết 3 lần
READPAST + version verify — thay thế Snapshot Isolation (không cần bật trên DB). Skip dirty rows + kiểm tra CT version consistency giữa đầu và cuối transaction.

2. CT change reader:

BEGIN TRANSACTION;  -- ReadCommitted
  @min_valid = CHANGE_TRACKING_MIN_VALID_VERSION(@object_id)
  if (@min_valid > @checkpoint) → throw CheckpointInvalidException
  @upper = CHANGE_TRACKING_CURRENT_VERSION()
  SELECT CT.*, P.*
  FROM CHANGETABLE(CHANGES ..., @checkpoint) CT
  LEFT JOIN ... AS P WITH (READPAST)
  WHERE CT.SYS_CHANGE_VERSION <= @upper
COMMIT;
CHANGETABLE: Table-valued function của SQL Server Change Tracking — không bị ảnh hưởng bởi DML transaction khác. READPAST chỉ áp dụng cho LEFT JOIN tới source table.

3. Full refresh reader:

SELECT {columns} FROM [dbo].[CRM.Partners] — không transaction, không hint. Gọi từ SyncOrchestrator path không dùng trong pilot hiện tại.

PostgresPartnersWriter PostgreSQL writer

Triển khai ISyncBatchApplier — ghi data + checkpoint trong 1 transaction

ApplyBootstrapAsync:

BEGIN
  -- Upsert all rows from bootstrap read
  INSERT INTO report.partners (...) VALUES (...)
    ON CONFLICT (partner_id) DO UPDATE SET ...

  -- Deactivate rows not in current bootstrap batch
  UPDATE report.partners SET is_active = false
  WHERE source_system = 'erp'
    AND partner_id != ALL(@snapshotPks)

  -- Set checkpoint = baseline version
  UPDATE sync_meta.checkpoint
  SET last_sync_version = @baselineVersion,
      sync_status = 'ready', ...
  WHERE source_table = @sourceTable
COMMIT
is_active = IsCustomer. COALESCE(EXCLUDED.is_customer, false) — partner chỉ active nếu là customer trên ERP.

ApplyBatchAsync (CT):

BEGIN
  -- Deactivate deleted rows (CT op = 'D')
  UPDATE report.partners SET is_active = false
  WHERE partner_id = ANY(@deletePks)

  -- Deactivate non-customer rows (CT op = 'I'/'U', IsCustomer != true)
  UPDATE report.partners SET is_active = false
  WHERE partner_id = ANY(@filterOutPks)

  -- Upsert active customer rows (CT op = 'I'/'U', IsCustomer == true)
  INSERT INTO report.partners (...) VALUES (...)
    ON CONFLICT (partner_id) DO UPDATE SET ...

  -- Advance checkpoint (optimistic guard)
  UPDATE sync_meta.checkpoint SET ...
  WHERE source_table = @sourceTable
    AND last_sync_version = @previousCheckpoint
  if (affected == 0) → ROLLBACK, return CONCURRENT_CHECKPOINT
COMMIT
Bootstrap deactivate: partner_id != ALL(@snapshotPks) — tất cả rows không có trong lần chạy bootstrap hiện tại sẽ bị deactivate. Khi không có row nào (0 partners), ALL rows bị deactivate (behavior đúng — không có partner nào trên ERP).
PostgresSyncCheckpointStore Checkpoint CRUD

Quản lý checkpoint — đọc, advance, transition to full resync

Methods:

MethodSQLMục đích
GetAsync()SELECT ... FROM sync_meta.checkpoint WHERE source_table = @tĐọc trạng thái hiện tại
AdvanceAsync()UPDATE ... SET last_sync_version = @next, last_success_at = NOW(), ... WHERE source_table = @t AND last_sync_version = @prevAdvance checkpoint với optimistic guard

Optimistic guard: AdvanceAsync chỉ update nếu last_sync_version == @previousCheckpoint. Nếu worker khác đã advance → 0 rows affected → rollback.

PostgresTableSyncLock Distributed lock

Per-table distributed lock — PostgreSQL advisory lock

Cách hoạt động:

-- Hash table name → 64-bit integer
@lockKey = FNV-1a("central-db-sync:{sourceTable}")

-- Try acquire (non-blocking)
SELECT pg_try_advisory_lock(@lockKey)
-- → true: lock acquired, trả về AdvisoryLockHandle
-- → false: lock busy, trả về null

-- Release khi handle disposed
SELECT pg_advisory_unlock(@lockKey)
MVCC scope: pg_try_advisory_lock là session-level lock. Lock chỉ được giải phóng khi connection đóng hoặc gọi unlock. Không liên quan tới PostgreSQL transaction.
FNV-1a hash: Deterministic 64-bit — cùng key luôn cho cùng hash. Không collision với hash của table khác vì prefix "central-db-sync:" tạo namespace riêng.

Dispose pattern:

await using var lockHandle = await syncLock.TryAcquireAsync() — C# await using đảm bảo lock handle luôn được dispose (unlock + close connection) khi ra khỏi scope, kể cả khi exception. Không cần try/finally thủ công.

VI. Lock & Transaction Analysis

Phân tích chi tiết cơ chế lock và transaction qua từng bước Concurrency & Isolation

1. PostgreSQL Advisory Lock (session-level)

Thuộc tínhGiá trị
Loại lockpg_try_advisory_lock(key) — non-blocking
ScopeSession-level (không transaction-scoped)
KeyFNV-1a hash of "central-db-sync:{sourceTable}"
Giải phóngawait using → Dispose → unlock + close connection
Mục đíchNgăn 2 worker sync cùng table đồng thời
Giới hạnKhông ngăn conflict cấp transaction (cần optimistic guard riêng)

2. Optimistic Guard trên Checkpoint

Phương thứcCách hoạt động
AdvanceAsync()UPDATE ... WHERE last_sync_version = @prevVersion
Nếu worker B advance trước, worker A update 0 rows → rollback
TransitionToFullResyncAsync()Không guard (force update)
→ Chỉ 1 worker active nhờ advisory lock, nên an toàn
ApplyBatchAsync()Kiểm tra affected == 0 → CONCURRENT_CHECKPOINT outcome

3. MSSQL Transaction Isolation

ComponentIsolationHint
Bootstrap readerReadCommittedWITH (READPAST)
CT readerReadCommittedWITH (READPAST) trên LEFT JOIN
Full refresh readerNone (no transaction)Không hint
READPAST skip rows đang bị lock bởi transaction khác (dirty rows). Không bị blocked, nhưng có thể miss rows. Kết hợp với CT version verify để đảm bảo consistency.

4. PostgreSQL Transaction (Apply)

ComponentTransactionIsolation
ApplyBootstrapAsync1 transaction (upsert + deactivate + checkpoint)ReadCommitted (default)
ApplyBatchAsync1 transaction (delete + deactivate + upsert + checkpoint)ReadCommitted (default)
SyncRunLog.WriteAsyncTransaction riêng (sau apply)ReadCommitted
Log ngoài transaction: WriteAsync nằm ngoài apply transaction. Nếu apply rollback nhưng log thành công → sync_run_log ghi nhận lỗi nhưng checkpoint không đổi. Đây là behavior đúng — log vẫn tồn tại để debug.

5. Concurrent Scenario Matrix

Kịch bảnPhòng chốngKết quả

Rủi ro: READPAST skip dirty rows → nếu ERP transaction quá lâu (dirty rows không commit kịp), snapshot có thể miss rows. Version verify sẽ bắt được nếu có concurrent DML nhưng không bắt được nếu dirty row là row duy nhất và transaction abort. Bootstrap retry sẽ handle nếu version mismatch.

IX. Summary & Gotchas

Tổng kết và lưu ý cho team maintain Key takeaways

Architecture decisions:

  • ⛔ Không dùng Snapshot Isolation — dùng ReadCommitted + READPAST + version verify (xem decision record)
  • 🔒 Advisory lock không transaction-scoped — lock được giải phóng khi connection close, không phải khi COMMIT/ROLLBACK
  • 🧪 Admin check đang disabled — cần re-enable trước UAT
  • 🔄 Retry 3 lần + exponential backoff — transient error sẽ retry với 1s, 2s, 4s delay
  • ⚠️ Bootstrap deactivate ALL rows không trong lần chạy hiện tại — nếu có 0 partners trên ERP, ALL rows ở PostgreSQL bị deactivate

Vận hành:

  • Monitoring: Health endpoint GET /health/CRM.Partners → Base44/Retool
  • Debug: sync_meta.sync_run_log có đủ history để trace
  • Recovery: Auto full resync khi CT retention cleanup
  • Thủ công: POST /bootstrap để force resync
  • Lock: pg_advisory_unlock nếu lock không release (connection leak)
  • Schedule: 1 phút — có thể tăng lên 5 phút nếu data không thay đổi nhiều
