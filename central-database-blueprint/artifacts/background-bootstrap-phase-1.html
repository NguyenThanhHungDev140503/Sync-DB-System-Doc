CORE-MOBILE-APP-Reporting — Background Bootstrap Phase 1

sync_meta.bootstrap_request · Durable work ticket + async job for manual bootstrap sync

I. Tổng quan

Phase 1

Mục tiêu

Cho phép admin kích hoạt bootstrap sync bất đồng bộ qua API. Trả về HTTP 202 ngay lập tức, Hangfire chạy BootstrapSyncService trong background. Trạng thái request được persist trong sync_meta.bootstrap_request và có thể query qua API status endpoint.

Design Decision

Tại sao có bảng bootstrap_request riêng?

Hangfire quản lý execution scheduling, còn sync_meta.bootstrap_request quản lý business workflow state:

  • Business states riêng: pending_enqueue, waiting_for_lock — không có trong Hangfire
  • Partial unique index đảm bảo một request active / table tại một thời điểm
  • requestId = runId: UUID dùng chung cho sync_run_log, tạo correlation trace
  • Watchdog: Per-request Hangfire scheduled job — recover request bị orphan (crash trước enqueue)
  • API Status: GET /bootstrap/{requestId} đọc từ store riêng, độc lập Hangfire
Kiến trúc tổng thể: Controller (WebApi) → BootstrapRequestService (Application) → PostgresBootstrapRequestStore (Infrastructure) + HangfireBootstrapJobScheduler → CentralDbSyncJobs.RunBootstrapAsync → BootstrapSyncService

II. Luồng hoạt động

2.1 Request Flow — Submit → Execution → Complete

Client → POST /api/central-db-sync/bootstrap/{table}
├─ CentralDbSyncController.TriggerBootstrap
│ ├─ 1. Kiểm tra RegisteredTables (hardcoded set)
│ └─ 2. Nếu không hợp lệ → 400 BadRequest
├─ BootstrapRequestService.SubmitAsync
│ ├─ requestStore.CreateOrGetActiveAsync
│ │ ├─ 1. SELECT active request → nếu có → return (IsNewRequest=false)
│ │ ├─ 2. INSERT + ON CONFLICT → nếu conflict → return existing
│ │ └─ 3. Mới → return PendingEnqueue (IsNewRequest=true)
│ ├─ [Mới] scheduler.ScheduleWatchdogAsync → Hangfire delayed 45s
│ ├─ (watchdog fires after 45s — checks still pending_enqueue → re-enqueue)
│ ├─ [Mới] scheduler.EnqueueAsync → Hangfire enqueue
│ │ └─ Hangfire gọi ngay CentralDbSyncJobs.RunBootstrapAsync
│ ├─ requestStore.MarkQueuedAsync → status = queued
│ │ └─ ⚠ Nếu job claim trước → MarkQueuedAsync fail (status running)
│ └─ requestStore.GetAsync → reload trạng thái mới nhất
├─ CentralDbSyncJobs.RunBootstrapAsync
│ ├─ requestStore.TryMarkRunningAsync → status = running
│ │ └─ Chấp nhận pending_enqueue, queued, waiting_for_lock
│ ├─ BootstrapSyncService.ExecuteAsync
│ │ ├─ [Lock OK] reader → applier → log → return Succeeded
│ │ ├─ [Lock Busy] return SkippedLocked
│ │ └─ [Error] log failed → return Failed (checkpoint unchanged)
│ └─ switch (Outcome)
│ ├─ Succeeded → SeedAsync(table_sync_config) → MarkCompletedAsync
│ │ └─ Seed fail → MarkFailedAsync(SeedConfigFailed)
│ ├─ SkippedLocked → MarkWaitingForLock → ScheduleAsync(1m) → MarkQueuedAsync
│ └─ default → MarkFailedAsync
└─ Client nhận HTTP 202 + statusUrl
└─ GET statusUrl → GetBootstrapStatus → HTTP 200 với trạng thái hiện tại 
// Application/Features/CentralDbSync/Services/BootstrapRequestService.cs:25-63
public async Task<BootstrapRequestResult> SubmitAsync(
    string sourceTable, CancellationToken ct)
{
    var result = await requestStore.CreateOrGetActiveAsync(sourceTable, ct);
    if (!result.IsNewRequest) return result;

    // 1. Schedule one-shot watchdog before primary enqueue
    await scheduler.ScheduleWatchdogAsync(sourceTable, requestId, 45s, ct);

    // 2. Enqueue the real bootstrap job
    var hangfireJobId = await scheduler.EnqueueAsync(sourceTable, requestId, ct);

    // 3. Mark queued — if still pending_enqueue (crash before this point, watchdog picks up)
    await requestStore.MarkQueuedAsync(requestId, hangfireJobId, ct);
    var updated = await requestStore.GetAsync(requestId, ct);
    return new BootstrapRequestResult(updated ?? result.Request, true);
}

VI. Race Conditions & Fixes

Critical

Race 1: Concurrent CreateOrGetActiveAsync

Vấn đề: Hai session gần như đồng thời gọi CreateOrGetActiveAsync cho cùng source_table.
Cả hai SELECT đều không tìm thấy active request → cả hai đều INSERT. Session thứ 2 gặp ON CONFLICT. Code cũ dùng ExecuteAsync trả về row count (luôn > 0 cho cả INSERT và ON CONFLICT DO UPDATE), nên session thứ 2 tưởng rằng INSERT thành công và trả về requestId của chính nó — nhưng request đó không tồn tại trong DB (phantom ID). Caller enqueue Hangfire job cho request không tồn tại → silent data loss.

Fix: Dùng QuerySingleAsync<Guid> đọc giá trị thực từ RETURNING request_id. So sánh với requestId đã generate — nếu khác nhau tức là conflict, fetch existing row.

// Infrastructure/CentralDbSync/PostgresBootstrapRequestStore.cs:48-97
// Trước (sai):
var inserted = await conn.ExecuteAsync(sql);
if (inserted > 0)  // ← luôn true, kể cả conflict!

// Sau (đúng):
var actualRequestId = await conn.QuerySingleAsync<Guid>(sql);
if (actualRequestId == requestId)
{
    // We inserted a new row
    return new BootstrapRequestResult(created, true);
}
// Conflict: fetch existing row by actualRequestId

Critical

Race 2: Job chạy trước MarkQueuedAsync

Vấn đề: BootstrapRequestService.SubmitAsync gọi scheduler.EnqueueAsync trước requestStore.MarkQueuedAsync. Hangfire có thể thực thi RunBootstrapAsync ngay lập tức. Job gọi TryMarkRunningAsync — nhưng request đang ở pending_enqueue, và code cũ chỉ chấp nhận queued/waiting_for_lock. Job silent exit. Sau đó MarkQueuedAsync set status thành queued. Request stuck ở queued vì reconciliation chỉ quét pending_enqueue, và không còn scheduled job nào để process nó.

Fix: Mở rộng TryMarkRunningAsync để chấp nhận pending_enqueue trong status filter.

// Trước:
AND status IN ('queued', 'waiting_for_lock')

// Sau:
AND status IN ('pending_enqueue', 'queued', 'waiting_for_lock')

📍 Infrastructure/CentralDbSync/PostgresBootstrapRequestStore.cs:153

Important

Race 3: MarkQueuedAsync không chấp nhận waiting_for_lock

Vấn đề: Trong SkippedLocked path: MarkWaitingForLockAsync (→ waiting_for_lock) → ScheduleAsync → MarkQueuedAsync. Code cũ chỉ cho phép MarkQueuedAsync từ pending_enqueue, nên fails vì request đang ở waiting_for_lock. Hậu quả: job ID mới không được persist, API show stale hangfireJobId và status waiting_for_lock.

Fix: Mở rộng MarkQueuedAsync để chấp nhận cả pending_enqueue và waiting_for_lock.

// Trước:
AND status = 'pending_enqueue'

// Sau:
AND status IN ('pending_enqueue', 'waiting_for_lock')

7.1 ExecuteAsync — Bootstrap Execution

BootstrapSyncService.ExecuteAsync
├─ 1. syncLock.TryAcquireAsync — advisory lock per table
│ ├─ [null] → log SkippedLocked, return
│ └─ [handle] → proceed
├─ 2. reader.ReadAsync — snapshot + CHANGE_TRACKING_CURRENT_VERSION
│ └─ [Exception] → catch → log → return Failed
├─ 3. applier.ApplyBootstrapAsync — upsert + deactivate + set checkpoint
│ └─ [Exception] → catch → log (checkpoint NOT modified) → return Failed
└─ 4. runLog.WriteAsync — ghi audit log success
└─ return Succeeded 
// Application/Features/CentralDbSync/Services/BootstrapSyncService.cs:24-57
public async Task<SyncRunResult> ExecuteAsync(
    TableSyncConfig config, Guid runId, CancellationToken ct)
{
    await using var lockHandle = await syncLock.TryAcquireAsync(config.SourceTable, ct);
    if (lockHandle is null)
    {
        await runLog.WriteAsync(SkippedLocked entry);
        return new SyncRunResult { Outcome = SkippedLocked };
    }
    return await ExecuteCoreAsync(config, runId, startedAt, ct);
}

7.2 Throw-caught Transaction Flow

SubmitAsync
├─ CreateOrGetActiveAsync
│ ├─ [Mới] → enqueue + mark queued → return
│ └─ [Existing] → return existing (no enqueue)
└─ [Exception]
└─ MarkFailedAsync (code = BootstrapEnqueueFailed) → rethrow

RunBootstrapAsync
├─ TryMarkRunningAsync
│ ├─ [Claimed] → execute bootstrap
│ └─ [Not claimed] → warning, exit
├─ BootstrapSyncService.ExecuteAsync
│ ├─ [Succeeded] → MarkCompletedAsync
│ ├─ [SkippedLocked] → MarkWaitingForLock → Schedule(1m) → MarkQueued
│ └─ [Failed] → MarkFailedAsync
└─ (retry handling: AutomaticRetry(Attempts=0) — internal retry via Schedule)

8.1 Ý tưởng

Không dùng poll (cron 5 phút). Thay vào đó, mỗi request tự schedule một one-shot watchdog ngay tại submit time. Watchdog chạy sau 45 giây. Nếu request đã được enqueue thành công trong khoảng đó → watchdog no-op. Nếu process crash trước enqueue → watchdog phát hiện request vẫn ở pending_enqueue → enqueue lại.

8.2 Sequence

SubmitAsync
├─ 1. INSERT pending_enqueue
├─ 2. ScheduleWatchdogAsync(delay=45s) → Hangfire schedule job
├─ 3. EnqueueAsync → Hangfire enqueue bootstrap job
├─ 4. MarkQueuedAsync → status = queued
└─ 5. Return 202

--- 45 giây sau ---
ReconcileBootstrapRequestAsync (watchdog fires)
├─ GetAsync(requestId)
├─ [status = queued/running] → no-op (healthy path)
└─ [status = pending_enqueue] → orphan detected!
├─ scheduler.ScheduleAsync(delay=0) → enqueue bootstrap job
├─ MarkQueuedAsync → queued
└─ log: "Watchdog reconciled orphan pending request"

// Application/Features/CentralDbSync/Services/BootstrapRequestService.cs:94-150
public async Task ReconcileOneAsync(string sourceTable, Guid requestId, CancellationToken ct)
{
    var request = await requestStore.GetAsync(requestId, ct);
    if (request is null) { return; }  // request deleted

    if (request.Status != BootstrapRequestStatus.PendingEnqueue)
    {
        // Already queued/running/completed — no-op
        return;
    }

    // Orphan detected — re-enqueue
    var hangfireJobId = await scheduler.ScheduleAsync(..., TimeSpan.Zero, ct);
    await requestStore.MarkQueuedAsync(request.RequestId, hangfireJobId, ct);
}

IX. PostgresBootstrapRequestStore — SQL Details

9.1 CreateOrGetActiveAsync — INSERT + ON CONFLICT Guard

-- Infrastructure/CentralDbSync/PostgresBootstrapRequestStore.cs:48-59
INSERT INTO sync_meta.bootstrap_request
    (request_id, source_table, status, rows_staged, total_rows_expected,
     attempt_count, requested_at, updated_at)
VALUES
    (@RequestId, @SourceTable, 'pending_enqueue', 0, NULL, 0, @Now, @Now)
ON CONFLICT (source_table)
    WHERE status IN ('pending_enqueue','queued','running','waiting_for_lock')
DO UPDATE SET updated_at = EXCLUDED.updated_at
RETURNING request_id
RETURNING request_id trả về ID thực tế trong DB (của row mới hoặc existing). So sánh với requestId đã generate để phân biệt INSERT thành công vs CONFLICT.

9.2 TryMarkRunningAsync — Atomic Claim

-- Infrastructure/CentralDbSync/PostgresBootstrapRequestStore.cs:147-156
UPDATE sync_meta.bootstrap_request
SET status = 'running',
    started_at = COALESCE(started_at, NOW()),
    attempt_count = attempt_count + 1,
    updated_at = NOW()
WHERE request_id = @RequestId
  AND status IN ('pending_enqueue', 'queued', 'waiting_for_lock')
attempt_count tăng mỗi lần claim — kể cả khi lock busy (SkippedLocked → waiting_for_lock → reschedule → claim lại). Cho phép operator thấy số lần retry.
