Nguyên tắc cốt lõi: Checkpoint là single source of truth cho trạng thái sync engine. Mọi kịch bản checkpoint hỏng đều dẫn về Bootstrap (full sync) để tự phục hồi. Engine không bao giờ stuck — luôn có đường thoát.

II. Trường hợp trông bất thường nhưng không phải checkpoint hỏng

Expected behavior

Nhiều table cùng tăng last_sync_version

Ví dụ: chỉ TableA có DML nhưng sau một sync cycle, checkpoint của TableA, TableB và TableC đều tăng từ 100 lên 101. Đây là behavior bình thường của SQL Server Change Tracking.

Nguyên nhân:

CHANGE_TRACKING_CURRENT_VERSION() → trả về current version của toàn database
CHANGETABLE(CHANGES TableX, @checkpoint) → chỉ trả event của TableX
ChangeTrackingSyncService → nếu 0 rows, vẫn gọi AdvanceAsync tới upper watermark 
-- Current checkpoint của cả hai table
TableA = 100
TableB = 100

-- Một thay đổi trên TableA làm database-wide watermark thành 101
Read TableA: events (100, 101] = 1 row → apply row → TableA checkpoint = 101
Read TableB: events (100, 101] = 0 row → no_changes → TableB checkpoint = 101
Ý nghĩa chính xác của checkpoint: TableB.last_sync_version = 101 không nói rằng TableB đã thay đổi ở version 101. Nó nói rằng sync engine đã kiểm tra TableB và xác nhận không còn CT event nào của TableB trong khoảng (100, 101]. Lần chạy sau có thể bắt đầu từ 101 mà không quét lại khoảng rỗng này.

Điều gì không xảy ra:

Nghi ngờThực tế
Một thay đổi trên TableA chạy SQL update tất cả checkpoint Không. Mỗi job chỉ update row có source_table = @sourceTable.
Checkpoint tăng khi không có row nghĩa là mất change Không. CHANGETABLE đã kiểm tra riêng table đó tới upper watermark trước khi advance.
Các checkpoint phải có version sequence độc lập Không. Chúng là cursor riêng theo table nhưng cùng tham chiếu một timeline version cấp database.
Chỉ cần điều tra khi: checkpoint của một table thay đổi mà không có sync run tương ứng; câu update thiếu filter source_table; checkpoint đi lùi; hoặc job advance qua upper watermark mà chưa query CHANGETABLE thành công. Việc nhiều table hội tụ về cùng version, tự nó, không phải lỗi.

III. 4 kịch bản checkpoint hỏng

Scenario 1

Checkpoint null / chưa ready

Xảy ra khi: seed lần đầu, checkpoint bị xóa tay, hoặc bootstrap trước set status = requires_full_resync

Luồng xử lý:

SyncOrchestrator.ExecuteAsync() → gọi checkpointStore.GetAsync()
Decision tree (dòng 58-62 SyncOrchestrator.cs): 
if (checkpoint is null
    || checkpoint.SyncStatus == "pending_initial_sync"
    || checkpoint.SyncStatus == "requires_full_resync")
    → BootstrapSyncService.ExecuteAsync()
else
    → ChangeTrackingSyncService.ExecuteAsync()

Chi tiết bootstrap path:

BootstrapSyncService.ExecuteAsync()
1. Acquire advisory lock (pg_try_advisory_lock)
2. IBootstrapSnapshotReader.ReadAsync() → đọc full snapshot từ MSSQL + CT baseline version
3. ISyncBatchApplier.ApplyBootstrapAsync() → 1 PostgreSQL transaction:
• Upsert tất cả rows từ snapshot
• Deactivate orphan rows (không còn trong snapshot)
• UPDATE checkpoint: set status = 'ready', version = baselineVersion
4. ISyncRunLog.WriteAsync() → ghi log (NGOÀI transaction apply)
5. Lock handle disposed → tự động unlock
📍 Atomic checkpoint invariant: step 3 là 1 transaction — data và checkpoint luôn consistent.
Kết quả: Checkpoint thành ready, last_sync_version có giá trị. CT incremental có thể chạy ở cycle sau.
Scenario 2

CT retention hết hạn — CheckpointInvalidException

Xảy ra khi: Sync downtime > 2 ngày (SQL Server CT retention default).
CHANGE_TRACKING_MIN_VALID_VERSION(object_id) > checkpoint.last_sync_version → MSSQL không còn dữ liệu CT cũ.

Đây là kịch bản phức tạp nhất — cơ chế same-lock recovery:

SyncOrchestrator → checkpoint ready → gọi ChangeTrackingSyncService.ExecuteAsync()
│ ├─ 1. Acquire advisory lock (pg_try_advisory_lock)
├─ 2. Validate checkpoint: ready + has version ✓
├─ 3. reader.ReadBatchAsync() → MSSQL CT query
│ └─ CHANGE_TRACKING_MIN_VALID_VERSION > checkpoint
│ → throw CheckpointInvalidException
│
├─ 4. catch (CheckpointInvalidException)
│ ├─ checkpointStore.TransitionToFullResyncAsync()
│ │ UPDATE checkpoint SET
│ │ sync_status = 'requires_full_resync'
│ │ consecutive_failure_count++,
│ │ last_error_code = 'CheckpointInvalid'
│ │ last_attempt_at = NOW()
│ │ last_failure_at = NOW()
│ │
│ └─ bootstrapService.ExecuteWithProvidedLockAsync()
│ ← KHÔNG acquire lock mới — chạy bootstrap core
│ với lock handle đã acquire từ bước 1
│
└─ 5. await using lockHandle → Dispose → unlock
Tại sao phải giữ lock xuyên suốt?
Nếu ChangeTrackingSyncService release lock ở bước 4 rồi acquire lại, worker khác có thể lấy lock ở giữa và bắt đầu sync table này. Khi worker hiện tại acquire lại, nó sẽ conflict với worker kia. ExecuteWithProvidedLockAsync giải quyết vấn đề này bằng cách dùng lại lock handle đã có.
Lưu ý: TransitionToFullResyncAsync set trạng thái error TRƯỚC khi bootstrap recovery chạy. Nếu recovery thành công, checkpoint được set lại ready. Nếu recovery thất bại, checkpoint giữ nguyên requires_full_resync → cycle sau SyncOrchestrator thấy error → gọi Bootstrap lại.

Source code — flow chính (ChangeTrackingSyncService.cs dòng 84-99):

catch (CheckpointInvalidException)
{
    logger.LogWarning("Checkpoint invalid for {SourceTable}");
    await checkpointStore.TransitionToFullResyncAsync(
        config.SourceTable, "CheckpointInvalid",
        "CT checkpoint is below minimum valid version", ct);

    return await bootstrapService.ExecuteWithProvidedLockAsync(
        config, startedAt, ct);
}
Scenario 3

Optimistic concurrency conflict — ConcurrentCheckpoint

Xảy ra khi: 2 worker cùng apply batch và cùng advance checkpoint (dù advisory lock đã ngăn, guard vẫn tồn tại để đề phòng key collision hoặc lock hết hạn bất thường).

Luồng:

PostgresPartnersWriter.ApplyBatchAsync()
│
├─ UPDATE sync_meta.checkpoint SET version = @next
│ WHERE source_table = @t AND last_sync_version = @previousCheckpoint
│
├─ affected > 0 → COMMIT, return succeeded
│
└─ affected == 0 → ROLLBACK, return RequiresFullResync
(có thể kèm error_code = CONCURRENT_CHECKPOINT)
ChangeTrackingSyncService nhận outcome = RequiresFullResync (dòng 145-152): 
if (result.Outcome == SyncStatus.Outcome.RequiresFullResync)
{
    await checkpointStore.TransitionToFullResyncAsync(
        sourceTable, result.ErrorCode, result.ErrorMessage, ct);
}
→ Cycle sau: SyncOrchestrator thấy requires_full_resync → gọi Bootstrap
Giải thích optimistic guard: WHERE last_sync_version = @previousCheckpoint là cơ chế phòng ngừa. Advisory lock ngăn 2 worker sync cùng lúc, nhưng nếu có lỗi lock (session timeout, unexpected release), guard này đảm bảo at-most-once semantics: nếu worker B đã advance checkpoint trước, worker A không thể ghi đè. Data vẫn consistent — chỉ bị "miss" 1 cycle sync, tự hồi phục ở cycle sau.
Scenario 4

Bootstrap chính nó bị fail

Xảy ra khi: MSSQL snapshot lỗi, network timeout, PostgreSQL connection loss, transient exception.

Luồng:

BootstrapSyncService.ExecuteCoreAsync()
│
├─ try: reader.ReadAsync() → applier.ApplyBootstrapAsync()
│
└─ catch (Exception ex): 
catch (Exception ex)
{
    // Ghi error metadata vào checkpoint
    await checkpointStore.TransitionToFullResyncAsync(
        config.SourceTable, "BootstrapFailed", ex.Message, CancellationToken.None);

    // Ghi audit log
    await runLog.WriteAsync(CreateFailedEntry(...));

    return new SyncRunResult
    {
        Outcome = SyncStatus.Outcome.Failed,
        ErrorCode = "BootstrapFailed",
        ErrorMessage = ex.Message
    };
}

Hậu quả và tự phục hồi:

Current cycle
• Checkpoint: requires_full_resync
• consecutive_failure_count + 1
• last_error_code = 'BootstrapFailed'
• sync_run_log: outcome = failed, errorCode = BootstrapFailed

Next cycle (1 phút sau)
• SyncOrchestrator đọc checkpoint → requires_full_resync → Bootstrap lại
• Nếu thành công → checkpoint = ready, failure_count reset về 0
• Nếu fail tiếp → consecutive_failure_count tăng → health Degraded
Rủi ro: Nếu Bootstrap fail liên tục (ví dụ: MSSQL connection down), consecutive_failure_count tăng mãi. Cần alert qua health endpoint. Không có giới hạn trên cho số lần retry — engine sẽ retry mãi mỗi phút.

IV. Server Crash Mid-Sync + ERP Data Change During Downtime

Vấn đề: Nếu sync job đang chạy giữa chừng thì server die, sau đó ERP có thay đổi data. Lúc server chạy lại, làm sao lấy được data trong khoảng downtime đó?

Câu trả lời: Không cần cơ chế đặc biệt. CT engine tự động xử lý nhờ (1) PG transaction atomic (checkpoint + data trong 1 tx), (2) Change Tracking accumulation, (3) Upsert idempotent.
Atomic

Cốt lõi: Checkpoint + data trong cùng 1 PG transaction

PostgresPartnersWriter.ApplyBatchAsync() — checkpoint advance và data writes là 1 atomic unit

-- PostgresPartnersWriter.cs dòng 45-132
BEGIN TX
  ├─ UPDATE report.partners SET is_active = false  -- xóa/deactivate
  ├─ INSERT INTO report.partners (...) VALUES (...)
  │     ON CONFLICT (partner_id) DO UPDATE SET ...  -- upsert
  └─ UPDATE sync_meta.checkpoint
        SET last_sync_version = @upperWatermark
        WHERE source_table = @sourceTable
          AND last_sync_version = @previousCheckpoint
COMMIT

Hệ quả: checkpoint chỉ advance sau khi data đã ghi thành công. Nếu crash giữa chừng, toàn bộ transaction rollback — checkpoint giữ nguyên.

Idempotent writes: ON CONFLICT (partner_id) DO UPDATE đảm bảo apply cùng 1 batch nhiều lần không gây duplicate hay corrupt data.
Scenario A

Server die TRƯỚC khi PG transaction commit

Ví dụ: đang upsert rows được 50% thì server tắt

Timeline:
1. CT reader đọc batch từ MSSQL (versions 11→20)
2. PG writer bắt đầu upsert → crash
└── PG transaction rollback (do connection die)
3. Checkpoint KHÔNG đổi — vẫn là version 10

Server restart → next sync cycle:
4. Đọc checkpoint — last_sync_version = 10
5. CT reader query: CHANGETABLE(CHANGES ..., 10)
6. CT trả về tất cả changes từ 10→current (gồm cả batch failed + changes mới từ ERP)
7. PG writer upsert lại — idempotent, không sao
8. Advance checkpoint lên version mới ✓
Kết luận: Không mất data. Checkpoint không đổi = engine re-process từ checkpoint cũ. Upsert làm phần còn lại.
Scenario B

Server die SAU KHI PG transaction commit

Ví dụ: PG commit xong, server tắt trước khi log được ghi hoặc process exit

Timeline:
1. CT reader đọc batch (versions 11→20)
2. PG writer upsert + commit + advance checkpoint → OK
3. Server crash sau commit

Server restart → next sync cycle:
4. Đọc checkpoint — last_sync_version = 20 (đã advance)
5. CT reader query: CHANGETABLE(CHANGES ..., 20)
6. Chỉ lấy changes từ 20→current — bỏ qua batch cũ ✓
Kết luận: Checkpoint đã advance, data đã ghi. Engine tiếp tục từ checkpoint mới. Không duplicate.
Scenario C

ERP thay đổi data trong thời gian server downtime

Đây là câu hỏi chính: trong lúc sync server down, ERP insert/update/delete — khi server chạy lại, làm sao lấy được các thay đổi đó?

Timeline với CT version:
10 ─ checkpoint (last_sync_version = 10)
11 ─ ERP update Partner A
12 ─ ERP insert Partner B
13 ─ ERP delete Partner C ← server đang down
14 ─ ERP update Partner D
...
25 ─ CHANGE_TRACKING_CURRENT_VERSION() = 25 ← server restart

Query thực tế (SqlServerPartnersReader.cs dòng 133-160):
SELECT ... FROM CHANGETABLE(CHANGES [CRM.Partners], 10) AS CT
LEFT JOIN [CRM.Partners] AS P ON P.PartnerId = CT.PartnerId
WHERE CT.SYS_CHANGE_VERSION <= 25
ORDER BY CT.SYS_CHANGE_VERSION, CT.PartnerId

→ Trả về tất cả rows với version 11→25
→ Bao gồm: update A, insert B, delete C, update D
→ Không thiếu thay đổi nào
Cơ chế: Change Tracking lưu lịch sử thay đổi độc lập với sync server. Dù sync server có down bao lâu, CT vẫn tiếp tục ghi nhận changes. Khi sync chạy lại, nó chỉ cần query từ checkpoint cũ — CT trả về tất cả pending changes, bao gồm cả những change phát sinh trong thời gian downtime.
Scenario D

Rủi ro duy nhất: CT retention hết hạn

Mặc định retention = 2 ngày. Nếu downtime > 2 ngày, CT cleanup old changes → checkpoint trở nên invalid. Đã được xử lý tại Scenario 2 ở trên.

Khi checkpoint < MIN_VALID_VERSION:
1. SqlServerPartnersReader throw CheckpointInvalidException
2. ChangeTrackingSyncService bắt exception
3. checkpointStore.TransitionToFullResyncAsync()
4. bootstrapService.ExecuteWithProvidedLockAsync() → full snapshot
5. Checkpoint mới = current CT version, status = ready

→ Tự động phục hồi, không mất data

Tổng kết tất cả kịch bản:

Kịch bảnCơ chếMất data?
Crash trước PG commitCheckpoint không đổi + upsert idempotentKhông
Crash sau PG commitCheckpoint đã advance, next run từ checkpoint mớiKhông
ERP change trong downtimeCT accumulation: query từ checkpoint cũ trả về tất cảKhông

V. State Machine — Chi tiết

┌────────────────────────────────────────────────────────────────────────────┐
│                         CHECKPOINT STATE MACHINE                           │
├────────────────────────────────────────────────────────────────────────────┤
│
    ── seed: INSERT INTO checkpoint (source_table, sync_status)
             VALUES ('CRM.Partners', 'pending_initial_sync')

    pending_initial_sync
         │
         │  Bootstrap thành công
         │  PostgresPartnersWriter.ApplyBootstrapAsync():
         │    UPDATE checkpoint SET
         │      last_sync_version = @baseline,      -- CT version đầu tiên
         │      sync_status = 'ready',
         │      last_success_at = NOW(),
         │      consecutive_failure_count = 0
         ▼
    ready
         │
         ├── CT sync thành công
         │   AdvanceAsync(): UPDATE SET version++, last_success_at, failure=0
         │
         ├── CT gặp lỗi (timeout, transient)
         │   → trả về Failed outcome
         │   → checkpoint KHÔNG đổi
         │   → cycle sau CT lại (vì status vẫn 'ready')
         │
         ├── CT apply retry hết 3 lần
         │   → trả về Failed (ApplyRetriesExhausted)
         │   → checkpoint KHÔNG đổi
         │
         ├── CT apply optimistic conflict
         │   → outcome = RequiresFullResync
         │   → TransitionToFullResyncAsync(): set status = 'requires_full_resync'
         │
         ├── CT invalid exception
         │   → TransitionToFullResyncAsync()
         │   → bootstrapService.ExecuteWithProvidedLockAsync()  -- immediate recovery
         │   → nếu OK: checkpoint = ready
         │
         └── Bootstrap fail (từ recovery path)
             → TransitionToFullResyncAsync() (failure_count++)
             → cycle sau retry

    requires_full_resync
         │
         │  Bootstrap thành công (từ SyncOrchestrator)
         │  → quay về ready
         │
         │  Bootstrap thất bại
         │  → giữ nguyên requires_full_resync
         │  → consecutive_failure_count++
         │  → cycle sau retry
         │
         ▼
    (loop đến khi thành công hoặc alert can thiệp)
└────────────────────────────────────────────────────────────────────────────┘
Nguyên tắc: Checkpoint chỉ giữ trạng thái error khi có lỗi thực sự. Lỗi transient (timeout, connection loss) KHÔNG làm thay đổi checkpoint — CT vẫn giữ 'ready' và retry ở cycle sau. Chỉ lỗi nghiêm trọng (CheckpointInvalid, ConcurrentCheckpoint, BootstrapFailed) mới transition sang requires_full_resync.

VI. Health Implications

Health endpoint (GET /api/central-db-sync/health/{sourceTable}) dùng checkpoint để quyết định trạng thái:

(dù status = 'ready')		 Degraded Có lỗi gần đây — thường tự hồi phục. Kiểm tra sync_run_log để trace
now - last_success_at > maxAllowedLag Degraded Sync không chạy hoặc bị trễ — có thể Hangfire job đã dừng

VII. Lock & Transaction — Race Condition Analysis

Kịch bảnPhòng chốngKết quả
2 workers bootstrap cùng lúc Advisory lock (pg_try_advisory_lock) Worker 2: skipped_locked
1 bootstrap + 1 CT cùng lúc Advisory lock (cùng key hash) Worker 2: skipped_locked
CT gặp CheckpointInvalid → release lock → acquire lại ExecuteWithProvidedLockAsync — giữ lock xuyên suốt Không conflict — lock ownership giữ nguyên
Bootstrap CT version drift (concurrent DML trên MSSQL) READPAST + verify-before-commit (compare CT version trước/sau) Rollback + retry (tối đa 3 lần) → ổn định
PostgreSQL apply optimistic conflict (2 workers, lock failure) WHERE last_sync_version = @previous guard affected == 0 → rollback → RequiresFullResync
Kết luận: Không có race condition nào dẫn đến data corruption. Trường hợp tệ nhất là sync bị skip 1 cycle (skipped_locked) hoặc cần resync (requires_full_resync). Cả 2 đều tự phục hồi ở cycle sau.
