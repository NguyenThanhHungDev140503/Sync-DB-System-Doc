Snapshot reader implementation detail

-- SqlServerPartnersReader.cs (bootstrap path)
-- Dùng ReadCommitted + READPAST thay vì Snapshot isolation
-- để không cần ALLOW_SNAPSHOT_ISOLATION trên DB
BEGIN TRANSACTION;  -- IsolationLevel.ReadCommitted
DECLARE @baseline bigint = CHANGE_TRACKING_CURRENT_VERSION();
SELECT {columns} FROM [dbo].[CRM.Partners] WITH (READPAST);
DECLARE @version_after bigint = CHANGE_TRACKING_CURRENT_VERSION();
-- Nếu baseline == version_after → commit, trả snapshot
-- Nếu khác → rollback, retry (tối đa 3 lần)
COMMIT TRANSACTION;
Verify-before-commit pattern. Đọc CT version → read data → re-check CT version. Nếu version không đổi, dữ liệu nhất quán. Nếu có thay đổi concurrent, rollback và retry. Loại bỏ hoàn toàn nhu cầu Snapshot Isolation trên DB.
sau bootstrap: checkpoint = ready

2. Validate checkpoint

CT reader query

DECLARE @upper bigint = CHANGE_TRACKING_CURRENT_VERSION();
SELECT CT.SYS_CHANGE_VERSION, CT.SYS_CHANGE_OPERATION,
       CT.PartnerId, P.*
FROM CHANGETABLE(CHANGES [dbo].[CRM.Partners], @checkpoint) CT
LEFT JOIN [dbo].[CRM.Partners] AS P WITH (READPAST)
    ON P.PartnerId = CT.PartnerId
WHERE CT.SYS_CHANGE_VERSION <= @upper
ORDER BY CT.SYS_CHANGE_VERSION, CT.PartnerId;
No changes optimization. Nếu batch.Rows.Count == 0, chỉ advance checkpoint tới UpperWatermark, không cần PostgreSQL data transaction.
Apply retry. ApplyWithRetryAsync retry tối đa 3 lần cho transient error (deadlock, timeout, connection). Exponential backoff: 1s → 2s → 4s.

Vì sao checkpoint của table không đổi dữ liệu vẫn tăng?

CHANGE_TRACKING_CURRENT_VERSION() là database-wide watermark. Nó biểu diễn vị trí hiện tại trên timeline Change Tracking của cả SQL Server database, không phải version riêng của table đang đọc. CHANGETABLE(CHANGES ..., @checkpoint) mới là phần lọc các event thuộc table cụ thể. 
-- Trạng thái ban đầu
TableA checkpoint = 100
TableB checkpoint = 100

-- Chỉ TableA có thay đổi; database-wide CT version tăng thành 101
TableA: CHANGETABLE(..., 100) → có rows  → checkpoint 101
TableB: CHANGETABLE(..., 100) → 0 rows  → checkpoint 101, outcome = no_changes
Không có mass update checkpoint. Job của mỗi table gọi AdvanceAsync(config.SourceTable, previous, upper) riêng, và PostgreSQL chỉ update row thỏa source_table = @sourceTable. Các table có thể hội tụ về cùng version vì chúng lần lượt xác nhận rằng không còn event của chính mình trước upper watermark đó.

5. PostgreSQL Apply Engine

06PostgresPartnersWriter

1 transaction — data changes + checkpoint advance

Bootstrap apply (ApplyBootstrapAsync):

-- 1 transaction:
BEGIN;
-- Upsert tất cả rows từ snapshot
INSERT INTO report.partners (...)
VALUES (...) ON CONFLICT (partner_id)
    DO UPDATE SET name = EXCLUDED.name, ...;

-- Deactivate rows không còn trong snapshot
UPDATE report.partners SET is_active = false
WHERE partner_id NOT IN (...snapshotPks...);

-- Set checkpoint = baseline version
INSERT INTO sync_meta.checkpoint (...)
VALUES (...) ON CONFLICT (source_table)
    DO UPDATE SET last_sync_version = @version,
                  sync_status = 'ready';
COMMIT;

CT apply (ApplyBatchAsync):

-- 1 transaction:
BEGIN;
-- Với mỗi changed row:
--   'D' → DELETE FROM report.partners WHERE partner_id = @id
--   'I'/'U' → INSERT ... ON CONFLICT DO UPDATE

-- Advance checkpoint
UPDATE sync_meta.checkpoint
SET last_sync_version = @upper_watermark,
    last_success_at = NOW(),
    consecutive_failure_count = 0
WHERE source_table = @source_table
  AND last_sync_version = @previous_version;
COMMIT;
Optimistic concurrency checkpoint. Update checkpoint với WHERE last_sync_version = @previous_version. Nếu 0 rows affected (race condition), rollback toàn bộ transaction. Đảm bảo at-most-once semantics: retry sẽ re-read changes từ version cũ.

6. Recovery Path

07CheckpointInvalidException

CT retention hết hạn — full resync immediate recovery

CT read
ReadBatchAsync()
→
CheckpointInvalid
min_valid > checkpoint
→
Transition state
TransitionToFullResyncAsync()
→
Bootstrap recovery
ExecuteWithProvidedLockAsync()
Lock ownership. Bootstrap recovery chạy trong cùng lock handle không release → re-acquire. Nếu release rồi acquire lại, worker khác có thể lấy lock ở giữa và gây conflict.

Error states được xử lý:

CT Sync Failed
(syntax error, connection loss, etc.)
→ Trả về SyncRunResult.Failed
→ Không advance checkpoint
→ Retry ở cycle sau
Bootstrap Failed
(snapshot isolation, lock wait, etc.)
→ TransitionToFullResyncAsync()
→ Log + checkpoint error metadata
→ Retry = full resync
Apply retry exhausted
(3 lần transient error)
→ outcome: failed
→ error_code: ApplyRetriesExhausted
→ Next CT cycle retry tự nhiên

09Lock & Health

Per-table advisory lock + health endpoint

PostgresTableSyncLock
Sử dụng pg_advisory_xact_lock với hash từ source_table name. Lock tự động release khi transaction kết thúc.

GET /api/central-db-sync/health/{sourceTable}
Return: status 0 (Healthy) nếu sync gần đây và lag < maxAllowedLag. status 1 (Degraded) nếu quá hạn hoặc có lỗi. status 2 (Unknown) nếu table không tồn tại.

9. Design Decisions & Deviations

D1Snapshot Isolation → READPAST

Vấn đề: Code gốc dùng IsolationLevel.Snapshot, yêu cầu ALTER DATABASE SET ALLOW_SNAPSHOT_ISOLATION ON trên MSSQL dev DB dùng chung. Team ngại ảnh hưởng các bảng khác.

Quyết định: Không bật Snapshot Isolation. Thay bằng ReadCommitted + READPAST + verify-before-commit retry (tối đa 3 lần). Xem docs/notes/2026-07-19-snapshot-isolation-decision.md.

D2Admin role guard

Vấn đề: POST /bootstrap yêu cầu HttpContext.User.IsInRole(AppConst.Admin). Test account manager@ua không có Admin role.

Quyết định: Tạm comment check để unblock testing. TODO re-enable trước UAT. Xem docs/notes/2026-07-19-admin-role-bypass.md.

D3CT Retention Recovery

Nếu sync downtime quá 2 ngày, checkpoint last_sync_version < CHANGE_TRACKING_MIN_VALID_VERSION → CheckpointInvalidException → tự động full resync recovery với ExecuteWithProvidedLockAsync giữ nguyên lock handle. Đây là behavior mong muốn.
